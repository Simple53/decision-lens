# 领域研究与决策支持工作流

## 核心定位

**AI 负责：** 明确硬性筛选条件、不限数量扫描并输出全量候选池 Markdown 表格、执行中英双语及中文互联网深挖、调用 `read_url_content` 提取真实 README/网页数据、在正文对比矩阵与评分表中保留 10–15 个核心方案、在 Phase 1 强制调用系统提问/暂停工具等待用户确认、编写方案对比矩阵与深度技术卡片。

**AI 默认不负责：** 推荐最佳方案、替用户做价值取舍。如果用户明确要求推荐，可以给出带理由的倾向性建议，但必须标注"这是基于当前证据的倾向性判断"。

---

## 铁律

1. **无限制候选池扫描与全量表格 (Unconstrained Pool & Full Table)**：
   - 候选池扫描绝不限制数量（尽可能扫描全网出现的项目）。
   - 在 Artifact 报告中，必须以类 Excel 格式的 Markdown 大表格输出**所有扫描到的候选项目全量清单**，列出项目名称、软件形态、Hard Constraint 否决/通过状态及判定原因。
2. **正文与矩阵保留 10–15 个核心方案 (Top 10–15 Core Candidates)**：
   - 从全量候选池中筛选并保留 **10–15 个最符合条件的核心方案**，深入展开至主方案维度对比表、加权评分表与深度技术剖析卡片中，避免只提供 3–4 个稀疏选项。
3. **中英双语与中文互联网深挖规程 (Bilingual & Chinese Community Scans)**：
   - 检索时必须同时使用英文与中文检索式（如 `开源 桌面 AI agent 编程 替代 Claude Code site:zhihu.com OR site:v2ex.com OR site:juejin.cn`）。
   - 必须主动覆盖知乎、V2EX、稀土掘金、B站及 Gitee 等中文开发者社区，挖掘国内开源桌面 Agent 项目与中文 API 配置实测。
4. **强制实地检索与网页读取规程 (Mandatory Live Search & Extraction Protocol)**：
   - **严禁凭 LLM 内部记忆脑补功能**。对每个选出的核心项目，必须显式执行独立的 `search_web` 检索，并对重点项目调用 **`read_url_content` 真正读取其 README / 官网页面** 提取最新功能、安装依赖与配置路径。
5. **硬性条件一票否决 (Hard Constraint Filtering)**：
   - 严格区分 **Hard Constraints（一票否决硬性指标，如“必须是独立桌面端”）** 与 **Soft Preferences（加权评估软性指标）**。
   - 不符合硬性条件的候选项目在全量表中标注 `[一票否决]`，且不占入主 10-15 个核心方案对比矩阵中。
6. **强制交互暂停等待确认 (Mandatory Interactive Pause)**：
   - 在 Phase 1 完成后，AI **必须调用 `ask_question` 工具**（或在对话框中明确停止工具调用），向用户弹出包含硬性限制、评估权重及动态信源清单的确认菜单。
   - 必须等待用户点击确认或补充修改意见后，方可启动 Phase 3 深度下钻检索。
7. **标准尾注与有效网页超链接系统 (Clean Endnote Citation System)**：
   - 正文中所有的关键事实、结论、项目特性、数据与指标，使用上角标 `[^1]`、`[^2]` 标记。
   - 文末设立 `## 参考文献与索引` (References) 章节，采用标准 Markdown 脚注格式（不加前面的多余 `-` 连字符）：
     ```markdown
     [^1]: [信源标题说明](https://valid-url.com) - 详细上下文描述
     ```
   - 所有 URL 必须是有效的真实 HTTP/HTTPS 网页超链接，绝不捏造虚假链接。
8. **移除装饰性 Note 提示块**：
   - 在生成的 Artifact 报告开篇，**不得添加 `> [!NOTE]` 等装饰性提示块**，直接进入报告标题与正文。
9. **Artifact 为主要载体**：创建 `domain_research_<主题>.md` 作为研究报告，对话框只输出简要摘要和需要用户决策的问题。
10. **评分诚实**：使用 1-5 整数定性评分，杜绝虚假小数精度。

## 阶段工作流 (Phases)

### Phase 1：研究边界、动态信源探索与参数锁定
明确 Hard Constraints 与 Soft Preferences；初步探索发现当前问题下最靠谱的 3-5 个信源平台（含中文社区）；**强制调用 `ask_question` 工具弹出确认菜单**，等待用户确认或修改需求。

### Phase 2：全量候选池扫描与领域地图
扫描 GitHub Topics, Gitee, 知乎, V2EX 等构建不限数量的全量候选池；输出全量项目 Markdown 大表格（标注否决/通过原因）；从通过项目中筛选出 10-15 个核心方案；构建认知地图，正文全量标注角标 `[^N]`。

### Phase 3：定向深挖与真实网页读取 (`read_url_content`)
在用户批准的信源范围内，对选出的 10-15 个核心方案进行独立的 `search_web` 检索并调用 `read_url_content` 提取 README；梳理冲突观点表；编写深度技术剖析卡片。

### Phase 4：10-15 核心方案矩阵与透明评分矩阵
构建 **10-15 个核心方案** 的维度对比表；构建权重透明表；生成 1-5 分定性加权评分表。

### Phase 5：决策交付与尾注索引
交付已知事实、未确定项与验证建议；输出条件决策路径提示；在文末输出不带 `-` 连字符的 `## 参考文献与索引` 真实网页超链接列表。
